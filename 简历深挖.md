# 2. 大厂算法岗最可能深挖的问题

## 2.1 总体简历问题

### Q1：请用 1 分钟介绍你的研究经历。

**建议回答：**

我的研究主线是多模态理解和工具增强型 Agent 系统。前期我做了 CFVBench，构建了一个面向细粒度 Video MRAG 的 benchmark，包含 599 个公开视频和 5360 个开放式问答，用来评估模型在长视频中检索和利用细粒度多模态证据的能力。这个项目让我发现现有 MLLM 的瓶颈并不只是生成能力，而是对短暂、细粒度、多模态证据的定位和利用能力不足。

此外，我还做过 Physic-HM，用物理过程到结果的单向依赖建模工业多模态异常检测，把低维传感器上下文注入高维音视频表征，在 Weld-4M 上达到 90.7% I-AUROC。

CFVBench 解决的是“怎么评估细粒度视频证据检索和生成”的问题；ReTool-Video 解决的是“怎么通过工具调用和递归 grounding 去获得这些证据”的问题；Physic-HM 则是在工业异常检测场景中研究“如何把物理先验和多模态表征结合”，本质上也关注多模态证据与决策之间的可解释关系。

所以三者并不是孤立的。它们都围绕一个核心问题：

> 多模态模型不能只依赖黑箱端到端生成，而需要更强的证据获取、结构化推理和可验证机制。

---

### Q3：你最有代表性的贡献是什么？

**建议回答：**

我认为最代表性的贡献是 ReTool-Video 中的 recursive grounding 和工具库设计。

现有 video agent 常见问题是：planner 会生成很抽象的动作，比如“find the relevant moment”或“analyze the object interaction”，但这些动作并不能直接执行。
**planner 的推理和工具之间存在语义鸿沟（semantic gap），直接让 planner 生成底层工具参数往往会失败；工具也很难做的完全符合 planner 的意图。**
传统方法要么把动作空间做得很粗，导致工具调用不可控；要么只用平面的 action representation，无法处理复杂工具链。

我的做法是把 action 分为两层：

1. **High-level intent**：planner 表达想要获得什么证据；
2. **Executable tool chain**：resolver 把 intent 落到具体工具、参数和执行顺序。

这样可以缓解 planner 直接生成底层 API 参数的困难，也能提高复杂任务中的工具组合能力。

# 3. ReTool-Video 深挖问题与标准回答

## 3.1 项目动机

### Q1：为什么需要 ReTool-Video？直接用 GPT-4o / Gemini / Qwen-VL 看视频不行吗？

**建议回答：**

直接把视频输入 MLLM 有几个问题：

1. **上下文预算有限**（已经被逐渐解决）
   长视频包含大量帧、音频、字幕、OCR、动作细节，模型无法完整接收所有原始信息。

2. **细粒度信息容易丢失**
   很多问题依赖短暂细节，例如某一帧中的数字、屏幕按钮、手部动作、字幕变化。均匀抽帧容易漏掉这些信息。

3. **模型缺少外部专用工具能力**
   OCR、ASR、object tracking、temporal localization、chart parsing 这些任务，专用工具往往比通用 MLLM 更可靠。

4. **缺少可验证证据路径**
   端到端回答只能给结果，很难判断答案来自哪个片段、哪个工具、哪条证据。

所以 ReTool-Video 的核心思想是：让大模型不直接“凭感觉看视频”，而是作为 planner，主动调用工具获取证据，再基于证据回答。

---

### Q2：你说 existing video agents 有 coarse tool spaces 和 flat action representations，具体是什么意思？

**建议回答：**

**Coarse tool space** 指工具粒度太粗。比如只有：

```text
caption_video(video)
answer_question(video, question)
retrieve_frames(video, query)
```

这种工具虽然简单，但它把很多不同能力混在一起，planner 很难精确控制。例如“找出视频中人物拿起红色杯子的时间段，并识别杯子上的文字”，这需要 object detection、temporal grounding、OCR、frame filtering 等多个子能力，不适合用一个粗粒度工具解决。

**Flat action representation** 指所有工具调用都在同一层动作空间中。planner 必须一次性决定：

```text
tool_name + all parameters + execution order
```

这对长视频任务太难，因为高层意图和底层工具参数之间存在 semantic gap。

ReTool-Video 通过 recursive grounding 把这个过程拆开：

```text
Question
  ↓
High-level intent
  ↓
Resolver
  ↓
Executable tool chain
  ↓
Evidence
  ↓
Final answer
```

这样 planner 只负责“想要什么证据”，resolver 负责“如何把证据获取过程落地”。

---

## 3.2 系统架构

### Q3：ReTool-Video 的整体架构是什么？

**建议回答：**

可以分为五个模块：

```text
User Question + Video
        ↓
Planner
        ↓
High-level Intent
        ↓
Resolver / Recursive Grounding
        ↓
Executable Tool Chain
        ↓
Tool Executor
        ↓
Evidence Memory
        ↓
Verifier / Answer Generator
```

各模块作用是：

1. **Planner**
   根据问题和已有 evidence 判断下一步需要什么信息。

2. **Resolver**
   把 planner 的抽象 intent 转换成具体工具调用，包括工具选择、参数修复、工具替换和工具链分解。

3. **Tool Executor**
   执行 OCR、ASR、video captioning、temporal localization、retrieval、rerank 等工具。

4. **Evidence Memory**
   保存每一步工具输出，包括时间戳、帧、文本、置信度、工具来源。

5. **Verifier / Answer Generator**
   检查证据是否足够、是否冲突，并生成最终答案。

---

### Q4：请形式化定义 ReTool-Video。

**建议回答：**

可以把视频问答任务定义为：

给定视频 $V$ 和问题 $q$，目标是生成答案 $a$，并找到支持答案的证据集合 $E$：

$$
a^* = \arg\max_a P(a \mid q, V, E)
$$

其中证据 $E$ 不是直接给定，而是由工具调用序列产生：

$$
E = {e_1, e_2, \dots, e_T}
$$

每一步工具调用为：

$$
e_t = \tau_t(V, q, h_{t-1}; \theta_{\tau})
$$

其中：

* $\tau_t$ 是第 $t$ 步选择的工具；
* $h_{t-1}$ 是之前的推理历史和 evidence；
* $\theta_{\tau}$ 是工具参数。

Planner 的策略可以写成：

$$
\pi_\theta(u_t \mid q, V, h_{t-1})
$$

其中 $u_t$ 是高层 intent。Resolver 将 $u_t$ 映射为可执行工具链：

$$
g_\phi(u_t) = [(\tau_1, p_1), (\tau_2, p_2), \dots, (\tau_k, p_k)]
$$

最终目标是最大化任务奖励：

$$
J(\theta) = \mathbb{E}*{\pi*\theta} \left[ R(a, E, q, V) \right]
$$

奖励可以包含答案正确性、工具调用成功率、证据充分性和成本惩罚：

$$
R = \lambda_1 R_{\text{ans}} + \lambda_2 R_{\text{tool}} + \lambda_3 R_{\text{evidence}} - \lambda_4 C_{\text{cost}}
$$

## 3.3 工具库设计

### Q5：为什么要构建 134 个工具？是不是太多了？

**建议回答：**

134 个工具不是为了堆数量，而是为了把视频理解中的能力拆成可组合的原子能力和 meta 能力。

我们将工具分为两类：

1. **Base tools：26 个**
   负责直接获取多模态证据，例如：

   * frame sampling
   * video captioning
   * OCR
   * ASR
   * object detection
   * temporal localization
   * action recognition
   * scene change detection

2. **Meta tools：108 个**
   负责对证据进行二次处理，例如：

   * filter
   * aggregate
   * rerank
   * deduplicate
   * format conversion
   * confidence calibration
   * evidence selection

这样做的好处是：

* base tools 提供感知能力；
* meta tools 提供推理前的数据组织能力；
* planner 不需要一次性处理所有原始输出；
* 工具链更可解释、可调试、可替换。

如果只有少量粗粒度工具，系统会缺乏精确控制；如果无限增加工具，planner 的 action space 会膨胀。所以关键不是数量，而是工具 schema、分层设计和 recursive grounding。

---

### Q6：工具 schema 如何设计？

**建议回答：**

一个工具 schema 至少需要包含：

```json
{
  "name": "ocr_frames",
  "description": "Extract text from selected video frames.",
  "input_schema": {
    "frame_ids": "List[int]",
    "language": "str",
    "region": "Optional[bbox]"
  },
  "output_schema": {
    "texts": "List[str]",
    "confidence": "List[float]",
    "frame_ids": "List[int]"
  },
  "cost": "float",
  "failure_modes": ["blurred frame", "small text", "occlusion"]
}
```

核心原则是：

1. **输入输出结构化**，便于后续工具消费；
2. **描述语义清晰**，便于 planner 选择；
3. **包含失败模式**，便于 resolver 替换工具；
4. **保留时间戳和置信度**，便于 evidence verification；
5. **支持组合**，一个工具输出能作为另一个工具输入。

---

### Q7：工具调用失败怎么办？

**建议回答：**

工具失败分为几类：

1. **参数错误**
   例如时间戳越界、frame id 不存在。
   解决：resolver 做参数修复。

2. **工具不适配**
   例如对无文字画面调用 OCR。
   解决：工具替换，比如改成 captioning 或 object detection。

3. **输出为空**
   例如 retrieval 没找到相关片段。
   解决：扩大时间窗口、调整 query、提高采样密度。

4. **输出冲突**
   例如 OCR 和 caption 结果不一致。
   解决：调用 verifier 或 reranker，保留多证据并让模型判断。

5. **成本过高**
   例如对全视频逐帧 OCR。
   解决：先 coarse retrieval，再局部 dense sampling。

## 3.4 Recursive Grounding

### Q8：recursive grounding 具体做了什么？

**建议回答：**

Recursive grounding 的目标是把 planner 的抽象 intent 落到可执行工具链。

例如问题是：

```text
What number appears on the screen right after the speaker mentions revenue growth?
```

Planner 可能产生 intent：

```text
Find the moment where the speaker mentions revenue growth, then inspect the screen text.
```

Resolver 会把它递归分解为：

```text
1. ASR(video) → transcript with timestamps
2. Search transcript for "revenue growth"
3. Extract local time window around matched timestamp
4. Sample frames densely in that window
5. OCR sampled frames
6. Rerank OCR outputs by temporal proximity
7. Return evidence
```

这里的关键是 planner 不直接写底层工具参数，而是由 resolver 完成：

```text
high-level intent
    → sub-intent
        → executable tool call
            → evidence
```

---

### Q9：为什么叫 recursive，而不是普通 decomposition？

**建议回答：**

普通 decomposition 通常是一次性把任务拆成几个步骤；recursive grounding 强调的是：如果某个子 intent 仍然不可执行，就继续向下分解，直到每一步都可以映射到工具库中的具体工具。

形式化地说，如果 $u$ 是一个 intent，判断它是否 executable：

$$
\text{Exec}(u) =
\begin{cases}
1, & u \in \mathcal{T} \
0, & u \notin \mathcal{T}
\end{cases}
$$

其中 $\mathcal{T}$ 是可执行工具空间。

如果 $\text{Exec}(u)=0$，则继续分解：

$$
u \rightarrow {u_1, u_2, \dots, u_k}
$$

直到：

$$
\forall i,\ \text{Exec}(u_i)=1
$$

这就是 recursive grounding。

---

### Q10：recursive grounding 和 chain-of-thought 有什么区别？

**建议回答：**

Chain-of-thought 主要是语言层面的推理过程，而 recursive grounding 是面向外部工具执行的结构化行动过程。

区别如下：

| 对比项  | Chain-of-thought | Recursive grounding              |
| ---- | ---------------- | -------------------------------- |
| 表达形式 | 自然语言推理           | intent 到工具链的映射                   |
| 是否执行 | 不一定执行            | 必须落到可执行工具                        |
| 目标   | 帮助模型推理           | 获取外部证据                           |
| 风险   | 可能编造推理           | 可通过工具输出验证                        |
| 输出   | reasoning text   | structured tool calls + evidence |

所以 recursive grounding 更接近 agent planning，而不是单纯 prompt 技巧。

## 3.5 Planner-side RL

### Q11：你说你做了 planner-side RL，具体 state、action、reward 是什么？

**建议回答：**

可以把 planner 看作一个策略模型 $\pi_\theta$。

### State

状态 $s_t$ 包含：

$$
s_t = (q, h_t, E_t, B_t)
$$

其中：

* $q$：用户问题；
* $h_t$：历史工具调用和推理轨迹；
* $E_t$：当前已获得证据；
* $B_t$：预算信息，例如剩余轮数、token、工具调用成本。

### Action

动作 $a_t$ 是 planner 生成的下一步高层 intent 或停止动作：

$$
a_t \in \mathcal{A} = {\text{intent}_1, \text{intent}_2, \dots, \text{stop}}
$$

例如：

```text
retrieve relevant temporal segments
inspect OCR in selected frames
verify answer consistency
stop and answer
```

### Transition

执行动作后，resolver 将其转成工具链，工具链返回新 evidence：

$$
s_{t+1} = f(s_t, a_t, \text{Tool}(a_t))
$$

### Reward

奖励设计为：

$$
R = \lambda_1 R_{\text{answer}}

* \lambda_2 R_{\text{tool}}
* \lambda_3 R_{\text{evidence}}
* \lambda_4 R_{\text{format}}

- \lambda_5 C_{\text{turn}}
- \lambda_6 C_{\text{tool}}
  $$

其中：

* $R_{\text{answer}}$：最终答案是否正确；
* $R_{\text{tool}}$：工具调用是否成功；
* $R_{\text{evidence}}$：证据是否支持答案；
* $R_{\text{format}}$：输出格式是否符合要求；
* $C_{\text{turn}}$：对话轮数成本；
* $C_{\text{tool}}$：工具调用成本。

---

### Q12：为什么不只用 supervised fine-tuning，而要用 RL？

**建议回答：**

SFT 只能模仿已有轨迹，但 agent 的工具调用存在探索问题：

1. 同一个问题可能有多条工具路径；
2. 有些工具链虽然不在人类标注轨迹中，但实际更高效；
3. 工具调用质量不能只看单步是否像人，而要看最终任务是否完成；
4. planner 需要学习何时继续调用工具、何时停止。

RL 更适合优化序列级目标：

$$
\max_\theta \mathbb{E}*{\tau \sim \pi*\theta} [R(\tau)]
$$

其中 $\tau$ 是完整工具调用轨迹：

$$
\tau = (s_1, a_1, r_1, \dots, s_T, a_T, r_T)
$$

SFT 优化的是 token-level likelihood：

$$
\max_\theta \sum_t \log p_\theta(y_t \mid y_{<t}, x)
$$

而 RL 优化的是最终任务回报，这更符合 agent 任务。

---

### Q13：RL 里面如何避免 reward hacking？

**建议回答：**

Reward hacking 的典型情况包括：

1. planner 为了提高 tool success rate，只调用简单工具；
2. planner 为了减少轮数，过早停止；
3. planner 为了 evidence reward，堆砌无关证据；
4. planner 学会迎合 judge，而不是真正解决问题。

解决方式：

1. **多目标 reward 平衡**
   不只看答案，也看证据充分性、工具成功率和成本。

2. **引入 verifier**
   检查 evidence 是否真的支持 answer。

3. **对无效工具调用加惩罚**
   如果工具输出为空或与问题无关，降低 reward。

4. **对过早停止加惩罚**
   如果 evidence 不足但直接回答，惩罚。

5. **人工 spot check**
   定期抽样检查高 reward 轨迹是否合理。

6. **分离训练 reward 和最终评测指标**
   防止模型只优化某个单一 reward proxy。

---

### Q14：PPO 的核心公式是什么？

**建议回答：**

PPO 的核心是限制新旧策略差异，避免策略更新过大。

概率比为：

$$
r_t(\theta)=\frac{\pi_\theta(a_t|s_t)}{\pi_{\theta_{\text{old}}}(a_t|s_t)}
$$

PPO clipped objective：

$$
L^{\text{CLIP}}(\theta)
=======================

\mathbb{E}_t
\left[
\min
\left(
r_t(\theta) A_t,
\text{clip}(r_t(\theta), 1-\epsilon, 1+\epsilon) A_t
\right)
\right]
$$

其中 $A_t$ 是 advantage：

$$
A_t = Q(s_t,a_t) - V(s_t)
$$

直观解释：

* 如果新策略和旧策略差别不大，就正常更新；
* 如果新策略变化过大，就 clip；
* 这样可以稳定训练，避免策略崩掉。

---

### Q15：GRPO 和 PPO 有什么区别？

**建议回答：**

PPO 通常需要 value model 来估计 advantage；GRPO 的思想是对同一 prompt 采样多个 response，用组内相对 reward 来估计 advantage，从而减少对 value model 的依赖。

假设对同一个问题采样 $G$ 条轨迹：

$$
{\tau_1, \tau_2, \dots, \tau_G}
$$

每条轨迹 reward 为 $r_i$，组内均值和标准差为：

$$
\mu = \frac{1}{G} \sum_{i=1}^G r_i
$$

$$
\sigma = \sqrt{\frac{1}{G}\sum_{i=1}^G (r_i-\mu)^2}
$$

则第 $i$ 条轨迹的相对 advantage 可写为：

$$
A_i = \frac{r_i-\mu}{\sigma}
$$

这样模型学习的是：同一个问题下，哪些工具调用轨迹相对更好。

对于 ReTool-Video 这种 agent 任务，GRPO 的优点是实现相对简单，尤其适合 final reward 或 trajectory-level reward 比较明确的场景。

---

## 3.6 实验与指标

### Q16：ReTool-Video 应该怎么评测？

**建议回答：**

我会从四个层面评测：

### 1. 最终回答准确率

$$
\text{Accuracy} = \frac{N_{\text{correct}}}{N_{\text{total}}}
$$

衡量最终答案是否正确。

### 2. 工具调用成功率

$$
\text{Tool Success Rate}
========================

\frac{N_{\text{successful tool calls}}}{N_{\text{all tool calls}}}
$$

但这里要注意：工具调用成功不等于任务成功。工具可能成功执行，但返回无关证据。因此还要有 evidence relevance。

### 3. 多轮任务完成率

$$
\text{Completion Rate}
======================

\frac{N_{\text{completed tasks}}}{N_{\text{tasks}}}
$$

衡量 agent 是否能在限定轮数内完成复杂任务。

### 4. 平均对话轮数 / 工具调用成本

$$
\text{Avg Turns}
================

\frac{1}{N}
\sum_{i=1}^N T_i
$$

同时还可以统计：

$$
\text{Avg Tool Cost}
====================

\frac{1}{N}
\sum_{i=1}^N \sum_{t=1}^{T_i} c_{i,t}
$$

理想情况不是无限调用工具，而是在较低成本下获得足够证据。

---

### Q17：你会设计哪些 ablation study？

**建议回答：**

我会做以下消融：

| 消融项                     | 目的                                     |
| ----------------------- | -------------------------------------- |
| w/o recursive grounding | 验证递归 grounding 是否提升可执行性                |
| w/o meta tools          | 验证 meta tools 是否帮助证据整理                 |
| w/o RL                  | 验证 RL 是否优于纯 prompt / SFT               |
| w/o verifier            | 验证证据验证对 hallucination 的抑制              |
| base tools only         | 验证工具组合能力                               |
| fixed frame sampling    | 验证 adaptive / tool-driven sampling 的作用 |
| no tool substitution    | 验证失败恢复机制                               |
| no cost penalty         | 看模型是否过度调用工具                            |

每个消融都应该对应一个明确假设，而不是为了凑表格。

---

### Q18：如果 ReTool-Video 准确率提升了，但工具调用次数也增加很多，怎么解释？

**建议回答：**

这说明系统可能通过更高成本换来了更高准确率。是否可接受取决于应用场景。

我会进一步分析：

1. 单位成本收益：

$$
\Delta_{\text{efficiency}}
==========================

\frac{\Delta \text{Accuracy}}{\Delta \text{Cost}}
$$

2. 是否存在过度调用工具；
3. 哪些问题类型最需要额外工具；
4. 是否可以通过 early stopping 或 confidence threshold 降低成本；
5. 是否可以通过 meta tools 对 evidence 进行压缩。

如果是在研究评测中，准确率提升有价值；如果是工业部署，就必须同时优化 latency、token cost 和 GPU cost。

# 4. CFVBench 深挖问题与标准回答

## 4.1 Benchmark 动机

### Q1：为什么要做 CFVBench？已有 Video QA benchmark 不够吗？

**建议回答：**

已有 Video QA benchmark 往往更关注 coarse-grained understanding，例如事件顺序、动作类别、场景理解。但真实的 Video MRAG 场景需要模型从长视频中检索并利用细粒度证据，例如：

* 新闻视频中的字幕和数据；
* 软件教程中的按钮和界面变化；
* 图表报告中的数值；
* 演讲视频中的某一页 slide；
* 短暂出现的 OCR 信息。

CFVBench 的目标不是单纯测“模型能不能看懂视频”，而是测：

```text
模型能不能从长视频中检索到正确证据，
并把这些证据用于开放式回答。
```

这更接近真实 Video RAG / Video Agent 应用。

---

### Q2：什么是 MRAG？和普通 RAG 有什么区别？

**建议回答：**

普通文本 RAG 是：

$$
q \rightarrow \text{text retriever} \rightarrow D_q \rightarrow \text{LLM answer}
$$

其中 $D_q$ 是检索到的文本文档。

Multimodal RAG 是：

$$
q \rightarrow \text{multimodal retriever} \rightarrow E_q \rightarrow \text{MLLM answer}
$$

其中 $E_q$ 可以包含：

$$
E_q = {I, V, A, T, O}
$$

* $I$：图像帧；
* $V$：视频片段；
* $A$：音频；
* $T$：字幕或 ASR；
* $O$：OCR / object / chart / metadata。

Video MRAG 更难，因为视频有时间维度：

$$
V = {f_1, f_2, \dots, f_T}
$$

检索不仅要找相关内容，还要找相关时间段：

$$
[t_s, t_e] = \arg\max_{[i,j]} \text{Rel}(q, V_{i:j})
$$

---

## 4.2 数据构建

### Q3：CFVBench 的数据构建流程是什么？

**建议回答：**

可以分为五步：

```text
1. 视频收集
2. 视频切分与多模态信息提取
3. QA 生成
4. 人工验证与修改
5. 质量控制与统计分析
```

具体来说：

1. **视频收集**
   选择高信息密度视频，例如 chart-heavy reports、news broadcasts、software tutorials。

2. **信息提取**
   对视频做 frame sampling、ASR、OCR、字幕抽取、场景切分。

3. **QA 构造**
   生成需要细粒度证据才能回答的问题，避免只靠常识回答。

4. **人工验证**
   检查答案是否唯一、证据是否存在、问题是否清晰。

5. **质量控制**
   去掉模糊问题、无证据问题、答案不稳定问题、过度依赖外部知识的问题。

---

### Q4：如何避免 benchmark 数据泄漏？

**建议回答：**

可以从几个层面控制：

1. **视频来源控制**
   使用公开视频但记录发布时间，尽量避免过度热门内容。

2. **问题设计控制**
   问题依赖视频内部细节，而不是互联网常识。

3. **答案验证**
   答案必须能在视频中找到证据。

4. **去重**
   对问题、答案、视频标题、字幕进行 near-duplicate 检查。

5. **模型闭卷测试**
   比较有无视频 evidence 的表现，如果无 evidence 也能答对，说明问题可能泄漏或太常识化。

---

## 4.3 检索评测

### Q5：Video MRAG 的 retrieval 怎么评测？

**建议回答：**

如果每个问题有 ground-truth evidence segment，可以用 temporal IoU：

$$
\text{IoU}
==========

\frac{|[t_s,t_e] \cap [\hat{t}_s,\hat{t}_e]|}
{|[t_s,t_e] \cup [\hat{t}_s,\hat{t}_e]|}
$$

如果检索返回 top-K 片段，可以用 Recall@K：

$$
\text{Recall@K}
===============

\frac{1}{N}
\sum_{i=1}^{N}
\mathbb{I}
\left[
\exists j \le K,\ \text{IoU}(\hat{s}_{i,j}, s_i) \ge \theta
\right]
$$

也可以用 MRR：

$$
\text{MRR}
==========

\frac{1}{N}
\sum_{i=1}^{N}
\frac{1}{\text{rank}_i}
$$

如果有 graded relevance，可以用 nDCG：

$$
\text{DCG@K}
============

\sum_{i=1}^{K}
\frac{2^{rel_i}-1}{\log_2(i+1)}
$$

$$
\text{nDCG@K}
=============

\frac{\text{DCG@K}}{\text{IDCG@K}}
$$

---

### Q6：为什么 retrieval 好不一定 final answer 好？

**建议回答：**

因为 Video MRAG 至少包含两个阶段：

```text
retrieval → reasoning / generation
```

检索好只说明模型找到了相关证据，但生成阶段仍可能失败：

1. 证据太长，模型没有关注关键部分；
2. 证据包含干扰信息；
3. 模型无法理解图表、OCR、动作细节；
4. 模型推理链错误；
5. 模型 hallucinate。

所以要同时报告：

* retrieval metrics；
* generation metrics；
* end-to-end answer accuracy；
* evidence utilization rate。

---

## 4.4 AVR 深挖

### Q7：Adaptive Visual Refinement 是什么？

**建议回答：**

AVR 是一个 plug-and-play 框架，用于解决均匀抽帧漏掉细粒度证据的问题。

普通方法可能固定采样：

$$
F = {f_1, f_{1+k}, f_{1+2k}, \dots}
$$

但如果关键证据只出现 1 秒，固定采样可能漏掉。AVR 的思想是：

1. 先粗粒度检索相关时间段；
2. 判断问题是否需要细粒度视觉信息；
3. 对相关时间段提高采样密度；
4. 必要时调用 OCR、object detector、chart parser 等外部工具；
5. 将 refined evidence 输入 MLLM。

可以写成：

$$
d_t =
\begin{cases}
d_{\text{high}}, & \text{if } \text{NeedRefine}(q, s_t)=1 \
d_{\text{low}}, & \text{otherwise}
\end{cases}
$$

其中 $d_t$ 是采样密度。

---

### Q8：AVR 和 ReTool-Video 的关系是什么？

**建议回答：**

AVR 可以看作 ReTool-Video 的前身或一个子思想。

CFVBench 发现问题：细粒度视频证据容易被漏掉。AVR 通过 adaptive sampling 和工具调用缓解这个问题。

ReTool-Video 更进一步，把这个过程 agent 化：

* AVR 更像一个固定策略的 refinement pipeline；
* ReTool-Video 让 planner 根据问题动态决定是否检索、是否加密采样、是否 OCR、是否 rerank、是否验证；
* ReTool-Video 还通过 recursive grounding 和 RL 优化工具调用策略。

所以两者关系是：

```text
CFVBench 暴露问题
        ↓
AVR 提供初步解决方案
        ↓
ReTool-Video 系统化为工具调用 Agent
```

# 5. Physic-HM 深挖问题与标准回答

## 5.1 项目动机

### Q1：为什么工业异常检测需要多模态？

**建议回答：**

工业制造过程中的异常不一定只体现在图像上。比如焊接场景中：

* 传感器可以反映电流、电压、速度等过程变量；
* 音频可以反映振动、爆裂声、摩擦声；
* 图像可以反映焊缝形态、火花、缺陷区域。

单一模态可能存在盲区：

```text
视觉正常，但声音异常；
声音正常，但传感器波动异常；
传感器异常，但最终表面缺陷尚不明显。
```

所以多模态融合可以提高异常检测的鲁棒性。

---

### Q2：你说 Process → Result dependency，具体是什么意思？

**建议回答：**

工业制造中存在自然的物理因果方向：

```text
process variables → manufacturing result
```

例如焊接时，电流、电压、速度等过程参数会影响熔池状态和最终焊缝质量。也就是说，传感器信号不是和视觉结果平行独立的模态，而是更接近“原因”或“过程条件”。

因此 Physic-HM 的核心不是简单 concat 多模态特征，而是用 sensor context 去调制 audio-visual representation。

形式化地说，给定 sensor feature $s$ 和 audio-visual feature $z_{av}$，我们希望学习一个调制函数：

$$
\tilde{z}*{av} = \gamma(s) \odot z*{av} + \beta(s)
$$

其中：

* $\gamma(s)$ 控制不同维度的放大或抑制；
* $\beta(s)$ 提供上下文偏移；
* $\odot$ 表示逐元素乘法。

这类似 FiLM modulation，但动机来自物理过程到结果的依赖关系。

---

## 5.2 多模态融合

### Q3：为什么不用简单 concat？

**建议回答：**

简单 concat 是：

$$
z = [z_v; z_a; z_s]
$$

它的问题是：

1. 不区分模态之间的因果关系；
2. 低维 sensor 特征容易被高维 audio-visual 特征淹没；
3. 对噪声模态不够鲁棒；
4. 无法表达 sensor 对视觉/音频特征的重要性调制。

Physic-HM 更合理的做法是：

$$
\tilde{z}*{av} = \text{Modulate}(z*{av}, z_s)
$$

再进行异常评分：

$$
S(x) = f_{\text{anom}}(\tilde{z}_{av})
$$

这样 sensor 不是被动拼接，而是主动调控 high-dimensional representation。

---

### Q4：异常检测的 anomaly score 怎么设计？

**建议回答：**

常见设计有三类：

### 1. Reconstruction-based

训练模型重建正常样本：

$$
\mathcal{L}_{rec} = |x - \hat{x}|_2^2
$$

异常分数：

$$
S(x) = |x - \hat{x}|_2
$$

异常样本重建误差更大。

### 2. Feature distance-based

学习正常特征中心 $c$：

$$
S(x) = |f(x)-c|_2^2
$$

越远离正常中心，越可能异常。

### 3. Density-based

估计正常分布 $p(x)$：

$$
S(x) = -\log p(x)
$$

低概率样本更可能异常。

Physic-HM 如果是 unsupervised，可以结合 feature distance、reconstruction 或 normality scoring。

---

## 5.3 指标

### Q5：什么是 AUROC / I-AUROC？

**建议回答：**

AUROC 是 ROC 曲线下面积。ROC 曲线横轴是 FPR，纵轴是 TPR：

$$
\text{TPR} = \frac{TP}{TP+FN}
$$

$$
\text{FPR} = \frac{FP}{FP+TN}
$$

AUROC 可以理解为：

> 随机抽一个正样本和一个负样本，模型给正样本更高异常分数的概率。

I-AUROC 通常指 image-level AUROC，也就是判断整张图或整个样本是否异常的 AUROC。

如果 I-AUROC = 90.7%，说明模型在样本级异常排序上表现较强，但它不一定代表 pixel-level localization 很好。因此如果有定位任务，还要报告 P-AUROC、PRO、AUPR 等指标。

---

### Q6：为什么 anomaly detection 不能只看 accuracy？

**建议回答：**

因为异常检测通常类别极不平衡。假设 99% 是正常样本，一个模型永远预测正常，也有 99% accuracy，但它完全检测不出异常。

所以更适合使用：

* AUROC：看排序能力；
* AUPR：更关注正类稀缺场景；
* FPR@95TPR：看高召回下误报率；
* PRO：看异常区域定位质量。

# 6. Human vs LLM Misinformation Detection 深挖问题

## Q1：这个项目的核心研究问题是什么？

**建议回答：**

核心问题是：

> 在 misinformation detection 中，人类和 LLM 的错误模式是否一致？LLM 是真的具备事实判断能力，还是只是依赖语言模式和表面线索？

我们构建了 ErrorQuestionDataset，包含 346 个 misinformation-related questions，并划分为五类错误。然后比较人类和五个代表性 LLM 在不同错误类别下的检测准确率和响应模式。

---

## Q2：如何保证人类和 LLM 比较公平？

**建议回答：**

需要控制以下变量：

1. **相同输入**
   人类和模型看到相同的问题和材料。

2. **统一输出格式**
   例如都输出 true/false 或多选答案，避免模型因格式自由获益。

3. **控制模型温度**
   评测时 temperature 应接近 0，减少随机性。

4. **多次采样或固定 seed**
   对模型输出稳定性进行检查。

5. **人类实验说明一致**
   避免不同受试者理解任务不同。

6. **统计显著性检验**
   不只报告平均准确率，还要报告置信区间或检验结果。

---

## Q3：可以用什么统计检验？

**建议回答：**

如果比较同一批题目上两个模型或人类与模型的正确/错误，可以用 McNemar test。

设：

|      |     B 正确 |     B 错误 |
| ---- | -------: | -------: |
| A 正确 | $n_{11}$ | $n_{10}$ |
| A 错误 | $n_{01}$ | $n_{00}$ |

McNemar 统计量：

$$
\chi^2 =
\frac{(|n_{10}-n_{01}|-1)^2}{n_{10}+n_{01}}
$$

它关注的是 A 对 B 错和 A 错 B 对的样本数量是否显著不同。

如果分析多类别错误分布，可以用 chi-square test。如果分析标注一致性，可以用 Cohen's kappa：

$$
\kappa =
\frac{p_o - p_e}{1-p_e}
$$

其中 $p_o$ 是观察一致率，$p_e$ 是随机一致率。

# 7. 八股知识点：你必须会答

## 7.1 Transformer / Attention

### Q1：Self-Attention 公式是什么？

**标准回答：**

输入 token 表示 $X$，通过线性变换得到：

$$
Q = XW_Q,\quad K = XW_K,\quad V = XW_V
$$

Attention 输出为：

$$
\text{Attention}(Q,K,V)
=======================

\text{softmax}
\left(
\frac{QK^\top}{\sqrt{d_k}}
\right)V
$$

其中 $\sqrt{d_k}$ 是缩放因子，防止 $QK^\top$ 数值过大导致 softmax 梯度过小。

---

### Q2：Multi-head Attention 为什么有用？

**标准回答：**

单头 attention 只能在一个子空间中建模关系，多头 attention 可以在多个子空间并行关注不同关系：

$$
\text{head}_i =
\text{Attention}(QW_i^Q, KW_i^K, VW_i^V)
$$

$$
\text{MHA}(Q,K,V)
=================

\text{Concat}(\text{head}_1,\dots,\text{head}_h)W^O
$$

在视频任务中，不同 head 可能关注：

* 物体关系；
* 动作变化；
* 时间顺序；
* 字幕信息；
* 局部视觉细节。

---

### Q3：LayerNorm 和 BatchNorm 区别？

**标准回答：**

BatchNorm 对 batch 维度统计均值方差：

$$
\mu_B = \frac{1}{m}\sum_{i=1}^{m}x_i
$$

适合 CNN，但依赖 batch size。

LayerNorm 对单个样本的 hidden dimension 统计：

$$
\mu = \frac{1}{d}\sum_{j=1}^{d}x_j
$$

不依赖 batch size，更适合 Transformer 和 NLP / LLM。

---

## 7.2 多模态对齐

### Q1：CLIP 类 contrastive learning 的损失是什么？

**标准回答：**

给定 batch 中 $N$ 对图文样本，图像 embedding 为 $v_i$，文本 embedding 为 $t_i$，相似度：

$$
s_{ij} = \frac{v_i^\top t_j}{\tau}
$$

image-to-text loss：

$$
\mathcal{L}_{i2t}
=================

-\frac{1}{N}
\sum_{i=1}^{N}
\log
\frac{\exp(s_{ii})}
{\sum_{j=1}^{N}\exp(s_{ij})}
$$

text-to-image loss：

$$
\mathcal{L}_{t2i}
=================

-\frac{1}{N}
\sum_{i=1}^{N}
\log
\frac{\exp(s_{ii})}
{\sum_{j=1}^{N}\exp(s_{ji})}
$$

总损失：

$$
\mathcal{L}
===========

\frac{1}{2}
(\mathcal{L}*{i2t}+\mathcal{L}*{t2i})
$$

直观上，就是让匹配的图文对相似度更高，不匹配的更低。

---

### Q2：视频和文本如何对齐？

**标准回答：**

视频可以表示为帧序列：

$$
V = {f_1, f_2, \dots, f_T}
$$

常见方法包括：

1. **Mean pooling**

$$
z_V = \frac{1}{T}\sum_{t=1}^{T} f_\theta(f_t)
$$

2. **Temporal Transformer**

$$
z_V = \text{Transformer}(f_1,\dots,f_T)
$$

3. **Segment-level representation**

$$
z_{V_i} = \text{Encode}(f_{s_i:e_i})
$$

4. **Query-aware pooling**

$$
\alpha_t =
\text{softmax}(q^\top W f_t)
$$

$$
z_V =
\sum_t \alpha_t f_t
$$

对 Video MRAG 来说，segment-level 和 query-aware representation 更适合，因为问题通常只依赖局部时间段。

---

## 7.3 RAG / Retrieval

### Q1：Dense retrieval 和 BM25 有什么区别？

**标准回答：**

BM25 是稀疏词项匹配，依赖关键词重合。其核心形式为：

$$
\text{BM25}(q,d)
================

\sum_{t \in q}
\text{IDF}(t)
\cdot
\frac{
f(t,d)(k_1+1)
}{
f(t,d)+k_1(1-b+b\frac{|d|}{avgdl})
}
$$

Dense retrieval 用神经网络把 query 和 document 映射到向量空间：

$$
s(q,d) = \text{sim}(E_q(q), E_d(d))
$$

常见相似度是 cosine：

$$
\cos(q,d)
=========

\frac{q^\top d}{|q||d|}
$$

BM25 适合精确词匹配，dense retrieval 适合语义匹配。实际系统常用 hybrid retrieval。

---

### Q2：Reranker 的作用是什么？

**标准回答：**

Retriever 通常为了速度，先召回 top-K 候选；reranker 用更强模型对候选重新排序。

流程是：

```text
Query
  ↓
Fast Retriever: top-K
  ↓
Cross-encoder / MLLM Reranker
  ↓
top-k refined evidence
```

Dense retriever 通常是 bi-encoder：

$$
s(q,d)=E_q(q)^\top E_d(d)
$$

Reranker 可以是 cross-encoder：

$$
s(q,d)=f_\theta([q;d])
$$

cross-encoder 更准，但成本更高。

## 7.4 RL 基础

### Q1：MDP 五元组是什么？

**标准回答：**

MDP 定义为：

$$
\mathcal{M} = (\mathcal{S}, \mathcal{A}, P, R, \gamma)
$$

其中：

* $\mathcal{S}$：状态空间；
* $\mathcal{A}$：动作空间；
* $P(s'|s,a)$：状态转移概率；
* $R(s,a)$：奖励函数；
* $\gamma$：折扣因子。

在 ReTool-Video 中：

* state 是问题、视频、历史工具调用和 evidence；
* action 是下一步 intent 或工具调用；
* reward 是答案正确性、工具成功率、证据质量、成本等。

---

### Q2：Policy Gradient 公式是什么？

**标准回答：**

目标函数：

$$
J(\theta)
=========

\mathbb{E}*{\tau \sim \pi*\theta}
[R(\tau)]
$$

梯度为：

$$
\nabla_\theta J(\theta)
=======================

\mathbb{E}*{\tau \sim \pi*\theta}
\left[
\nabla_\theta \log \pi_\theta(\tau) R(\tau)
\right]
$$

由于：

$$
\log \pi_\theta(\tau)
=====================

\sum_t \log \pi_\theta(a_t|s_t)
$$

所以：

$$
\nabla_\theta J(\theta)
=======================

\mathbb{E}
\left[
\sum_t
\nabla_\theta \log \pi_\theta(a_t|s_t)
R(\tau)
\right]
$$

为了降低方差，通常引入 baseline：

$$
\nabla_\theta J(\theta)
=======================

\mathbb{E}
\left[
\sum_t
\nabla_\theta \log \pi_\theta(a_t|s_t)
A_t
\right]
$$

其中：

$$
A_t = R_t - b(s_t)
$$

---

## 7.5 评测与统计

### Q1：如何判断一个方法提升是否显著？

**标准回答：**

不能只看平均值差异，还要看置信区间和统计检验。

对于 accuracy，可以计算标准误：

$$
SE =
\sqrt{
\frac{p(1-p)}{n}
}
$$

95% 置信区间近似为：

$$
p \pm 1.96 \cdot SE
$$

如果是同一批样本上比较两个模型，可以用 paired bootstrap 或 McNemar test。

对于多次运行的实验，可以报告：

$$
\mu \pm \sigma
$$

或者：

$$
\mu \pm \frac{1.96\sigma}{\sqrt{n}}
$$

   如何把自然语言高层意图转化为可执行工具链？

   如何用 RL 或反馈信号优化 planner 的多步工具调用策略？

## Q2：你的工作和已有 VideoAgent / VideoRAG / VideoMultiAgents 相比有什么区别？

**建议回答：**

   VideoRAG 更强调长视频检索增强生成；ReTool-Video 更强调工具调用链和递归 grounding。

2. **从固定 pipeline 到动态规划**
   很多方法依赖固定流程，例如先 caption 再 QA。ReTool-Video 中 planner 会根据问题动态选择工具和证据获取路径。

   我们不是让 planner 直接输出底层 API，而是先输出 high-level intent，再由 resolver 递归落地为可执行工具链。

4. **从 prompt-based agent 到 RL-optimized planner**
   我参与了 planner-side RL，使系统不只是靠 prompt，而是通过 trajectory-level reward 优化工具调用策略。

# 9. 大厂算法岗最可能深挖的问题

## Q1：如果把 ReTool-Video 部署到线上，你怎么设计系统？

**建议回答：**

线上系统可以分成：

```text
API Gateway
  ↓
Task Manager
  ↓
Planner Service
  ↓
Tool Router
  ↓
Tool Workers
  ↓
Evidence Store
  ↓
Verifier / Answer Service
  ↓
Logging & Monitoring
```

关键设计：

1. **异步工具执行**
   OCR、ASR、captioning 等耗时工具可以异步并行。

2. **缓存机制**
   对视频 embedding、ASR、OCR、frame features 做 cache。

3. **成本控制**
   给每个任务设置 max turns、max tool calls、max tokens。

4. **失败恢复**
   工具失败自动 retry、fallback 或 tool substitution。

5. **监控指标**

   * answer accuracy；
   * tool success rate；
   * latency P50/P95/P99；
   * average tool calls；
   * cost per query；
   * hallucination rate。

---

## Q2：如何降低长视频处理延迟？

**建议回答：**

可以从四层优化：

1. **预处理层**
   预先抽帧、ASR、OCR、embedding，避免查询时全量处理。

2. **检索层**
   先 coarse retrieval，再在候选片段内 dense processing。

3. **工具层**
   并行执行独立工具，例如 ASR search 和 frame retrieval 可以并行。

4. **模型层**
   使用小模型做 routing，大模型只处理关键 evidence。

整体策略是：

```text
cheap model filters first,
expensive model reasons later.
```

---

## Q3：如何判断 agent 是否 hallucinate？

**建议回答：**

可以设计 evidence-grounded verification：

1. 答案中的每个关键 claim 都要绑定 evidence；
2. verifier 判断 claim 是否被 evidence 支持；
3. 如果 evidence 不足，要求 agent 继续调用工具；
4. 如果 evidence 冲突，输出不确定性而不是强行回答。

形式化地说，对答案中的 claim 集合：

$$
A = {c_1, c_2, \dots, c_n}
$$

每个 claim 都需要证据支持：

$$
\forall c_i,\ \exists e_j \in E,\ \text{Support}(e_j, c_i) > \theta
$$

如果不满足，就判定为 unsupported claim。

# 11. 最后给你一套“高压追问—短答版”

## ReTool-Video

**问：一句话说创新点。**
答：把视频 Agent 的高层意图和底层工具执行解耦，通过 recursive grounding 把 planner intent 递归落地为可执行工具链，并用 RL 优化 planner 的多步工具使用策略。

**问：为什么不是普通 tool calling？**
答：普通 tool calling 让模型直接选择 API；ReTool-Video 增加 resolver，把不可执行的高层 intent 修复、替换或分解成工具链，更适合复杂视频任务。

**问：为什么需要 meta tools？**
答：base tools 产生原始多模态证据，meta tools 负责过滤、聚合、重排和格式化。没有 meta tools，planner 会面对大量噪声 evidence。

**问：RL reward 怎么设计？**
答：由答案正确性、工具成功率、证据支持度、格式正确性和成本惩罚组成：

$$
R = \lambda_1 R_{\text{ans}} + \lambda_2 R_{\text{tool}} + \lambda_3 R_{\text{evi}} - \lambda_4 C
$$

**问：最大失败模式是什么？**
答：检索不到关键片段、工具输出噪声大、planner 过早停止、工具链过长导致成本高，以及 verifier 无法判断隐含视觉细节。

---

## CFVBench

**问：CFVBench 和普通 VideoQA benchmark 区别？**
答：普通 VideoQA 多关注粗粒度理解；CFVBench 关注 Video MRAG 中的细粒度证据检索与开放式回答，尤其是短暂 OCR、图表、教程界面和新闻字幕等高密度信息。

**问：AVR 的核心思想？**
答：先粗检索相关片段，再对关键片段加密采样，并按需调用 OCR、ASR 等工具，从而减少固定抽帧漏掉细节的问题。

**问：如何证明 benchmark 有价值？**
答：通过系统评估 7 种 retrieval 方法和 14 个 MLLM，展示现有模型在 transient fine-grained multimodal details 上存在普遍失败，并通过 AVR 证明这个瓶颈可以被针对性缓解。

---

## Physic-HM

**问：为什么 sensor-guided modulation 比 concat 好？**
答：concat 假设模态平等并列，而工业制造中 sensor 更像 process condition。用 sensor 调制 audio-visual feature 更符合 Process → Result 的物理依赖。

**问：I-AUROC 90.7% 说明什么？**
答：说明模型在样本级异常排序上表现强，但还需要结合 pixel-level 或 localization 指标判断定位能力。
